# 串口设备接入与数据读取能力

## 1. 任务定位

本任务采用 **设计 + MVP 实现** 的方式推进，先落一个后端最小可用版本，把“串口接入、持续读取、自动重连、统一总线输出”跑通，再为 UI、统计、解析器和 MCP 链路预留稳定边界。

## 2. 已锁定决策

- 运行形态：先做后端 MVP，UI 后接。
- 首版输入格式：按行文本为主，同时总线保留原始字节字段，为后续二进制协议升级留口。
- MVP 连接管理：支持手动选择串口、配置波特率、建立连接和进程级断开。
- 自动重连：覆盖设备拔插、端口异常关闭和读错误场景；读超时不视为断线。
- 数据流向：先进入统一消息总线，供后续 UI / 统计 / 解析器消费。
- 发送能力：补充最小串口写入能力，并提供回环测试命令验证 TX/RX 闭环。

## 3. MVP 架构

### 3.1 数据流

```text
CLI Config
  -> Serial Open
  -> Line Framer
  -> Key:Value Parser
  -> Unified Message Bus
  -> Stdout Consumer
  -> Future Parser / Stats / UI

CLI Send
  -> Serial Write
  -> Optional Loopback Readback
```

### 3.2 模块边界

- `src/cli.rs`：处理 `ports` / `run` 命令和串口参数。
- `src/app.rs`：负责连接生命周期、自动重连和停止控制。
- `src/serial.rs`：负责串口枚举、打开串口、读取字节流、按行分帧。
- `src/serial.rs`：同时负责最小写入与回环读回验证。
- `src/key_value_parser.rs`：负责首版 `key:value` 结构化提取。
- `src/message.rs`：定义统一消息契约和连接状态模型。
- `src/bus.rs`：负责广播消息给多个消费者。

### 3.3 统一消息契约

- 所有数据都使用同一个 `BusMessage` 封装。
- 连接态使用 `ConnectionState` 表示：`Idle` / `Connecting` / `Connected` / `WaitingRetry` / `Stopped`。
- 数据态使用 `LinePayload` 表示，包含：
  - `text`：按行文本内容
  - `raw`：原始字节（不包含换行）
- 预留 `ParserMeta`，后续解析层可以在总线下游补充结构化字段。
- 当前 MVP 会在 `ParserMeta` 中输出：`channel_id`、`value`。

## 4. In Scope

- 串口枚举能力
- 手动指定端口和波特率启动读取
- 手动指定端口和波特率发送文本数据
- 单端口 TX->RX 回环验证命令
- 按行读取串口文本数据
- 最小 `key:value` 结构化解析
- 统一消息总线广播
- 自动重连监督循环
- 基础测试：分帧、总线广播、重连状态、`key:value` 解析

## 5. Out of Scope

- 更复杂的协议解析（JSON、二进制帧、CRC 校验、多字段正则）
- 串口写入和命令下发
- 多设备并发调度
- 历史数据持久化
- UI、MCP、统计服务的完整对接

## 6. 验收标准

- [ ] 可通过命令列出当前可用串口。
- [ ] 可通过命令指定串口和波特率启动读取。
- [ ] 可通过命令指定串口和波特率发送测试载荷。
- [ ] 在 TX 与 RX 短接时，可通过回环测试命令验证发送后被本端接收。
- [ ] 串口按行数据可被读入并输出到统一消息总线。
- [ ] `key:value` 文本可被提取为标准化字段并附着到总线消息。
- [ ] 总线消息同时包含时间戳、串口来源信息、连接状态或行数据负载。
- [ ] 端口打开失败或读取异常时，系统会进入等待重连状态并自动重试。
- [ ] 纯读取超时不会触发误重连。
- [ ] 关键状态行为具备独立测试覆盖。

## 7. 最小必要知识（MVK）

- 串口读取本质上是“字节流”，不是天然一条一条消息，所以必须先做分帧。
- 自动重连不能把“暂时没数据”当成“设备断了”，否则会频繁抖动。
- 统一总线的价值是把“接入层”和“消费层”解耦，后面接 UI 或 AI 不用重写读串口逻辑。
